[추가 지시: 계층별 upstream 통합 검증 체크포인트 생성]

이번 작업을 decompose할 때, feature spec들을 의존성 계층(layer)으로 정렬하라.
그리고 각 계층 경계마다 통합 검증 체크포인트 spec을 하나씩 추가하라.
통합 검증을 맨 끝에 하나만 두지 마라.

구성 원칙:
- feature spec을 의존성 순서로 배열하고, 어떤 spec들이 같은 계층(서로 독립, 동일 upstream에 의존)인지 식별한다.
- 각 계층이 완성되는 지점마다 "integration-check-L{n}" spec을 삽입한다.
  이 체크포인트는 해당 시점까지 완성된 upstream 집합 "전체"를 대상으로 삼는다
  (이번 계층만이 아니라, 그 아래 모든 층의 누적 결합을 검증).
- 따라서 뒤쪽 체크포인트일수록 검증 범위가 넓어지는 누적 구조가 된다.

각 체크포인트 spec에 담을 것:
- 검증 기준은 공유 계약 단일 소스(인터페이스/데이터 스키마/API 계약)이며, 개별 spec의 design이 아니라 이 단일 소스에 대해 대조한다.
- 이번 계층에서 새로 추가된 경계에 초점을 둔 cross-spec 시나리오를 명시하되, 아래 계층과의 결합도 함께 성립하는지 확인한다.
- mock 없이 실제 구현을 결합하는 integration/e2e 테스트로 구성한다.

roadmap.md에 반드시 기록할 것:
- 구현 순서: [계층1 feature spec들] → integration-check-L1 → [계층2 feature spec들] → integration-check-L2 → ... 순으로 배치한다.
  각 체크포인트는 바로 위 계층 impl 착수의 선행 조건이다.
- 게이트: integration-check-L{n}이 통과하기 전에는 계층 {n+1}의 impl을 금지한다.
- 재검증 트리거: 어떤 계층의 upstream spec이 수정되면, 그 계층 이후의 모든 체크포인트(L{n} 및 그 이상)를 누적 집합 기준으로 다시 실행해야 한다.

체크포인트 spec은 feature 로직을 구현하지 않는다. 오직 해당 시점까지의 upstream 누적 집합에 대한 계약·경계 정합 검증과 그 테스트만 범위로 삼는다.
