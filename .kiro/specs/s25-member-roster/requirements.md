# Requirements Document

## Introduction

s25-member-roster는 워크스페이스 owner(및 admin override)가 자신이 소유한 워크스페이스의 **현재 멤버 로스터**(이미 멤버인 사용자 전체)를 서버에서 권위 있게 열거할 수 있게 하는 기능이다. 이는 계약 공백 **S1**(멤버 목록 조회 GET 엔드포인트 부재)을 정식 해소한다.

배경: 현재 프론트 `MemberManagementPanel`의 멤버 목록은 로컬 세션 뮤테이션 상태(`useMemberActions().members`)로만 채워진다. 이 배열은 이번 브라우저 세션에서 추가/변경/제거로 확인된 멤버만 담기 때문에, 재로그인(=새 세션)이면 비어 있어 owner가 기존 멤버를 전혀 볼 수 없다. 또한 기존 멤버 뮤테이션 응답 스키마에는 이름이 없어 화면은 추가 시점에 이름을 별도로 캡처하는 우회에 의존한다.

이 기능은 (1) 백엔드에 owner 게이팅·admin override·anti-enumeration을 갖춘 멤버 로스터 조회 엔드포인트를 추가하고, (2) 프론트에서 마운트/재로그인 시 서버 로스터로 멤버 목록을 시드해 표시하며, 세션 뮤테이션 결과와 단일 소스로 통합한다. 멤버 뮤테이션 자체(추가/변경/제거)는 기존 spec 소유이며 이 기능은 소비만 한다.

## Boundary Context

- **In scope**:
  - 워크스페이스 멤버 로스터 조회 엔드포인트(owner 게이팅·admin override·anti-enumeration·전체 멤버 노출·결정적 순서).
  - 로스터 응답: 각 멤버의 `user_id`·이름·이메일·`role`을 담는 narrow 읽기 모델(민감 필드 비노출).
  - 프론트 로스터 조회 어댑터 + 로드 훅(`useWorkspaceMembers`) + `MemberManagementPanel`의 서버 로스터 시드 표시.
  - 서버 로스터와 로컬 세션 뮤테이션 상태의 단일 소스화(표시 일관성).
- **Out of scope**:
  - 멤버 뮤테이션(추가/역할변경/제거)의 백엔드·프론트 동작 — s05·s23 소유를 재사용만 한다.
  - 사용자 계정 생명주기(등록·비활성·삭제·재활성) — s03 소유.
  - assignable-users(멤버가 될 수 있는 후보) 조회 — s23 완료·소유.
- **Adjacent expectations**:
  - 접근 통제는 공용 워크스페이스 owner 게이트(`require_ws_role(OWNER)`)와 권한 resolver의 admin override(INV-1·2·3)를 재사용한다 — 이 기능은 게이트 판정 로직을 재구현하지 않는다.
  - 페이지 봉투(`Page`)와 narrow 읽기 모델 규약은 공유 계약(s01)과 s23 assignable 조회의 규약을 따른다.
  - 프론트는 공통 API 클라이언트(전역 401 인터셉터·baseURL·credentials)와 화면의 owner 게이팅(멤버십 role 소스)을 재사용한다.

## Requirements

### Requirement 1: owner의 멤버 로스터 조회

**Objective:** 워크스페이스 owner로서, 나는 내 워크스페이스의 현재 모든 멤버를 서버에서 권위 있게 조회하고 싶다. 그래야 재로그인 이후에도 기존 멤버를 빠짐없이 보고 관리할 수 있다.

#### Acceptance Criteria
1. When owner가 자신이 소유한 워크스페이스의 멤버 로스터를 조회하면, the 멤버 로스터 서비스 shall 성공 응답으로 그 워크스페이스의 현재 멤버 집합을 반환한다.
2. The 멤버 로스터 서비스 shall 각 멤버 항목에 대해 `user_id`, 이름, 이메일, `role`을 포함한다(이메일 값이 없는 멤버는 이메일이 비어 있을 수 있다).
3. The 멤버 로스터 서비스 shall 조회를 요청한 owner 자신을 포함하여 해당 워크스페이스의 모든 멤버십 항목을 로스터에 포함한다.
4. While 멤버 수가 단일 조회 페이지 크기를 초과하더라도, the 멤버 로스터 서비스 shall owner가 전체 멤버를 조회할 수 있도록 전체 멤버 집합을 노출하고 전체 멤버 개수를 함께 제공한다.
5. Where 멤버의 사용자 계정이 비활성 또는 삭제 상태이더라도, the 멤버 로스터 서비스 shall 그 멤버십 항목을 `role`과 함께 로스터에 포함한다.
6. When 동일한 워크스페이스와 동일한 조회 파라미터로 로스터를 반복 조회하면, the 멤버 로스터 서비스 shall 결정적이고 안정적인 순서로 멤버를 반환한다.

### Requirement 2: 접근 통제 · anti-enumeration · 최소 노출

**Objective:** 워크스페이스 owner로서, 나는 멤버 로스터가 오직 권한 있는 요청자에게만 노출되고 민감 정보나 워크스페이스 존재 여부가 새지 않기를 원한다. 그래야 통제된 폐쇄형 환경의 기밀성이 유지된다.

#### Acceptance Criteria
1. If 로스터 조회 요청에 유효한 인증 세션이 없으면, then the 멤버 로스터 서비스 shall 401로 거부한다.
2. If owner가 아닌 멤버(editor 또는 viewer)가 로스터 조회를 요청하면, then the 멤버 로스터 서비스 shall 403으로 거부한다.
3. If 해당 워크스페이스의 멤버가 아닌 사용자가 로스터 조회를 요청하면, then the 멤버 로스터 서비스 shall 403으로 거부한다.
4. If 존재하지 않는 워크스페이스에 대해 로스터 조회를 요청하면, then the 멤버 로스터 서비스 shall 워크스페이스의 존재 여부를 드러내지 않고 403으로 거부한다(404로 응답하지 않는다).
5. Where admin 세션이 로스터 조회를 요청하면, the 멤버 로스터 서비스 shall 요청자가 해당 워크스페이스의 owner인지와 무관하게 조회를 허용한다(admin override, INV-3).
6. The 멤버 로스터 서비스 shall 로스터 응답에서 로그인 식별자, 비밀번호 자격증명, 계정 상태 플래그, 내부 타임스탬프 등 로스터 표시에 불필요한 필드를 노출하지 않는다.

### Requirement 3: 프론트 로스터 로드와 화면 표시

**Objective:** 워크스페이스 owner로서, 나는 멤버 관리 화면에 진입할 때 서버의 현재 로스터가 자동으로 로드되어 표시되기를 원한다. 그래야 새 세션에서도 기존 멤버를 즉시 확인할 수 있다.

#### Acceptance Criteria
1. When 멤버 관리 화면이 마운트되거나 대상 워크스페이스가 변경되면, the 멤버 로스터 로더 shall 서버에서 해당 워크스페이스의 멤버 로스터를 조회한다.
2. When 사용자가 재로그인하여 새 세션으로 멤버 관리 화면에 진입하면, the 멤버 로스터 로더 shall 이전 세션의 로컬 뮤테이션 이력과 무관하게 서버 로스터로 멤버 목록을 시드한다.
3. While 로스터 조회가 진행 중이면, the 멤버 관리 화면 shall 로딩 상태를 표시한다.
4. If 로스터 조회가 실패하면, then the 멤버 관리 화면 shall 오류 상태를 표시한다.
5. When 로스터 조회가 성공하고 멤버가 존재하지 않으면, the 멤버 관리 화면 shall 빈 상태를 표시한다.
6. While 대상 워크스페이스가 선택되지 않았으면, the 멤버 로스터 로더 shall 조회를 수행하지 않고 안정적인 비로딩 상태를 유지한다.
7. The 멤버 관리 화면 shall 각 멤버의 이름을 서버 로스터가 제공한 값으로 표시하며, 추가 시점에 이름을 별도로 캡처하는 우회에 의존하지 않는다.

### Requirement 4: 서버 로스터와 세션 뮤테이션의 단일 소스화

**Objective:** 워크스페이스 owner로서, 나는 서버에서 로드된 로스터와 이번 세션에서 내가 수행한 멤버 변경이 하나의 일관된 목록으로 보이기를 원한다. 그래야 관리 화면이 항상 신뢰할 수 있는 단일 멤버 뷰를 제공한다.

#### Acceptance Criteria
1. When owner가 이번 세션에서 멤버를 추가하거나 역할을 변경하거나 제거하면, the 멤버 관리 화면 shall 서버로 시드된 로스터에 그 변경을 일관되게 반영한다(중복 없이 추가, 제거된 멤버 제외, 역할 변경 반영).
2. The 멤버 관리 화면 shall 서버 로스터와 세션 뮤테이션 결과를 단일 멤버 목록으로 통합하여 표시한다(상충하는 두 목록을 분리하여 두지 않는다).
3. When 사용자가 로스터를 다시 로드하면, the 멤버 로스터 로더 shall 목록을 서버의 현재 상태로 재동기화한다.
