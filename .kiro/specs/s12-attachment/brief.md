# Brief: s12-attachment

## Problem
문서에 붙는 이미지(붙여넣기)·첨부 파일을 파일로 저장하고 워크스페이스별 격리해야 한다. 문서
완전삭제나 저장으로 인한 참조 소멸 시 파일은 삭제하지 않고 "삭제된 파일 보관 폴더"로 이동한다.

## Current State
s01의 attachment 스키마(file_path·kind·is_archived·workspace_id) 존재. s07 문서·s09 버전·s10 완전삭제
흐름 확보. 파일 저장/격리/아카이브 동작 미구현.

## Desired Outcome
- 편집 중 이미지 붙여넣기 → 파일로 저장·문서에서 참조(REQ-8.1, base64 인라인 아님).
- editor+ 파일 첨부 → 저장·문서 연결(8.2).
- 파일 저장·보관 폴더 모두 WS별 격리(8.3·8.8).
- 문서 deleted(완전삭제) → 연결 첨부를 삭제하지 않고 보관 폴더 이동 `is_archived=true`(8.6).
- 저장으로 과거 버전만 참조하게 된 이미지 → 보관 폴더 이동(8.7).
- 보관 = 영구삭제 간주, 앱상 복원 대상 아님, admin 포함 조회 불가(8.9·8.10, INV-7). 단조 증가 수용(8.11).

## Approach
s07 문서·s09 버전(참조 소멸 판정)·s10 완전삭제 흐름 위에 첨부 서비스를 얹는다. 8.6은 완전삭제
반응으로, 8.7은 저장 반응으로 attachment가 소유(상위 계층이 아래 계층에 얹히는 방향). WS별 격리
디렉터리는 s01 config의 저장 루트 기준.

## Scope
- **In**: 이미지 붙여넣기 저장, 파일 첨부, WS 격리 저장, 완전삭제 시 보관 이동(8.6), 참조 소멸 시
  보관 이동(8.7), 보관 폴더 격리·비노출.
- **Out**: 링크 경유 파일 접근·차단(8.4·8.5는 s14 공유가 소비), 문서 상태 전이(s07·s10).

## Boundary Candidates
- 첨부/이미지 저장·격리
- 파일 생명주기(참조됨 → 보관 폴더 이동)

## Out of Boundary
- 공유 링크로의 파일 서빙·차단(8.4·8.5) → s14-sharing
- bundle 전이/완전삭제 로직(s07·s10) — 반응만 여기 소유

## Upstream / Downstream
- **Upstream**: s07(문서), s09(버전·참조 소멸 판정), s10(완전삭제 이벤트), s05(WS 격리), s11(게이트 통과)
- **Downstream**: s14-sharing(링크 경유 파일 접근에 저장·격리 결과 사용), s13 체크포인트

## Existing Spec Touchpoints
- **Extends**: 없음(신규). s09 버전·s10 완전삭제 흐름에 반응 로직 추가.
- **Adjacent**: s14-sharing(파일 접근 소비자)

## Constraints
물리 삭제 없음(INV-4, 보관 이동만), INV-7(보관=복원 불가), WS 격리(8.3·8.8), 파일 저장 루트는 config.yml. 한국어.
