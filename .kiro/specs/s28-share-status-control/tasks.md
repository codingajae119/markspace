# Implementation Plan

- [ ] 1. 백엔드 공유 상태 조회 엔드포인트 (읽기 전용)
- [x] 1.1 공유 상태 조회 서비스 메서드 추가 (읽기 전용)
  - 기존 단건 조회로 문서의 공유 링크(최대 1개)를 로드해 있으면 읽기 응답, 없으면 "링크 없음"을 나타내는 None 반환
  - 상태 전이 협력자(발급·전환·무효화·retire)를 호출하지 않으며 게이트·문서 status 를 관측조차 하지 않음
  - 단위 테스트: 링크 존재 시 token·is_enabled·share_url 포함 응답 / 링크 없음 시 None / 호출 전후 링크 행·토큰·활성 상태 불변 확인
  - _Requirements: 1.1, 1.2, 1.3_
  - _Boundary: ShareLinkService_

- [x] 1.2 공유 상태 조회 GET 라우트 추가 (발급·전환과 동일 게이트 재사용)
  - 발급·전환과 동일한 문서→워크스페이스 게이트(member 이상, admin bypass)를 붙인 읽기 전용 GET 라우트 노출
  - 링크 없을 때 200 + 본문 null, 링크 있을 때 200 + 공유 링크 응답으로 직렬화(발급·토글 응답과 균질)
  - 통합 테스트: 미인증 401 / 비멤버·비admin 403 / 문서 부재 404 / member·admin 200(링크 또는 null) — 발급·전환과 동일 게이트 동작 확인
  - _Requirements: 1.1, 1.4, 1.5, 1.6, 1.7_
  - _Depends: 1.1_
  - _Boundary: SharingRouter_

- [ ] 2. 프론트 공유 상태 조회 API·훅 확장 (sharing feature)
- [x] 2.1 (P) 공유 상태 조회 API 래퍼 추가
  - 공용 API 클라이언트로 문서 공유 상태 GET 요청을 보내는 타입 래퍼 추가(기존 발급·전환 래퍼 무변경)
  - 200 + null 본문 → null 반환, 오류(401/403/404)는 정규화된 API 오류로 전파하며 교차 관심사(인증·오류 정규화·전역 401)는 재구현하지 않음
  - 계약 테스트: GET 경로·메서드 결선 / null 본문 → null 반환 / API 오류 전파 확인
  - _Requirements: 2.1, 6.3_
  - _Boundary: shareApi_

- [x] 2.2 공유 관리 훅에 마운트 시 초기 조회 추가
  - 문서 선택·변경 시 초기 조회로 공유 상태를 시드하고 조회 in-flight 동안 loading 상태를 노출(확정 라벨 금지)
  - latest-wins 가드로 연속 문서 전환 시 가장 최근 선택 문서의 결과만 반영하고 이전(stale) 응답은 무시
  - 초기 시드가 재발급 통일 판정(INV-8)을 오염시키지 않음 — reissued 를 절대 set 하지 않고 이후 발급의 사전 링크 판정만 정직하게 반영
  - 조회 실패 시 오류를 표면화하되 링크 상태를 침범하지 않음(불확실한 상태를 공유 중으로 단정 금지), 기존 발급·전환·pending 시맨틱 보존
  - 단위 테스트: 초기 조회 1회·link 시드 / loading 전이 / latest-wins stale 무시 / 실패 시 error 표면화·link 불변 / reissued 격리 확인
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 6.1_
  - _Depends: 2.1_
  - _Boundary: useShareManager_

- [x] 2.3 기존 공유 테스트 재검증 갱신
  - 마운트 시 초기 조회가 추가됨에 따라 기존 공유 관리 훅·패널 테스트의 API 모크에 조회 응답 흡수를 추가
  - 기존 발급·전환·무효화 경로가 무회귀임을 확인 — 전체 sharing 테스트 스위트 통과
