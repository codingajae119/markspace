# Requirements Document

## Introduction

문서 공유 기능은 백엔드(s14-sharing)와 프론트(s22-fe-sharing)에 부품이 모두 구현되어 있으나, 두 결함으로 인해 사용자에게 도달하지 못한다: (1) 프론트 공유 관리 UI가 문서 화면에 마운트되어 있지 않아 공유 진입점이 없고, (2) 백엔드에 문서의 현재 공유 상태를 조회하는 수단이 없어 페이지 로드 시 "이 문서가 지금 공유 중인가"를 판별할 수 없다.

본 스펙은 문서 공유 상태를 조회하는 읽기 전용 백엔드 엔드포인트를 추가하고, owner/admin에게 문서 화면의 편집·삭제 컨트롤 옆에서 현재 공유 상태에 따라 "공유"(미공유 시) 또는 "공유 해제"(공유 중 시) 단일 토글 버튼을 노출하며, 공유 중일 때 게스트용 링크와 복사 수단을 함께 제공하는 것을 목표로 한다. 공유 링크 도메인 로직(토큰 발급, 재발급 통일 원칙, 무효화, 공개 렌더)은 재구현하지 않고 기존 구현에 위임한다.

## Boundary Context

- **In scope**:
  - 문서의 현재 공유 상태를 조회하는 읽기 전용 백엔드 엔드포인트(발급·전환 없음).
  - 문서 로드 시점의 공유 상태 초기 판별(프론트).
  - owner/admin + 공유 가능 워크스페이스 + active 문서 조건에서만 공유 상태 기반 단일 토글 버튼 노출.
  - 공유 중일 때 "공유 해제" 버튼 옆의 "링크 복사" 버튼(게스트용 링크를 클립보드로 복사).
- **Out of scope**:
  - 공유 링크 도메인 로직 재구현(토큰 발급/재발급, 재발급 통일 원칙(INV-8), 무효화 스윕, 공개 렌더·링크 경유 첨부 서빙).
  - DB 스키마 변경·마이그레이션.
  - 워크스페이스 공유 가능 게이트(is_shareable) 자체를 켜고 끄는 설정 UI.
- **Adjacent expectations**:
  - 서버측 권한 강제(문서→워크스페이스 게이트, member 이상 + admin bypass, 부재 404 / 미인증 401 / 비멤버 403)와 공유 불가·비active 시 409 거부는 기존 sharing 도메인이 소유한다. 본 스펙의 프론트 노출 게이팅은 UX 편의일 뿐 서버 강제를 대체하지 않는다.
  - 백엔드 공유 상태 조회 엔드포인트의 권한 게이트는 발급·전환과 동일하게 member 이상을 허용하되(admin bypass 포함), 프론트 컨트롤은 owner/admin에게만 노출한다(더 좁은 UX 정책). 두 계층의 기준 차이는 의도된 것이다.
  - 공개 렌더/무효화 판정, 공유 URL 규약(`/public/{token}`), 게스트용 프론트 링크 조립은 기존 구현에 위임한다.
  - 요청·인증·오류 정규화·전역 401 인터셉트 등 교차 관심사는 공용 API 클라이언트가 단일 소유한다.
  - 현재 워크스페이스의 role·공유 가능 여부(isShareable) 신호는 기존 워크스페이스 컨텍스트에서 취득한다.

## Requirements

### Requirement 1: 공유 상태 조회 엔드포인트 (백엔드)
**Objective:** 공유 관리 권한자로서, 문서의 현재 공유 상태를 조회하고 싶다. 그래야 로드 시점에 공유 여부를 판별할 수 있다.

#### Acceptance Criteria
1. When 인증된 member 이상 사용자가 특정 문서의 공유 상태를 요청하면, the 공유 상태 조회 API shall 해당 문서의 현재 공유 링크 상태(활성 여부·token·게스트용 공유 URL 포함)를 반환한다.
2. If 대상 문서에 공유 링크가 존재하지 않으면, then the 공유 상태 조회 API shall 오류가 아니라 "링크 없음"을 나타내는 정상 응답을 반환한다.
3. The 공유 상태 조회 API shall 조회 시 공유 링크를 발급·전환·무효화하지 않는다(읽기 전용).
4. If 대상 문서가 존재하지 않으면, then the 공유 상태 조회 API shall 발급·전환 엔드포인트와 동일하게 404를 반환한다.
5. If 요청자가 미인증이면, then the 공유 상태 조회 API shall 401을 반환한다.
6. If 요청자가 대상 워크스페이스의 member 미만이고 admin도 아니면, then the 공유 상태 조회 API shall 403을 반환한다.
7. The 공유 상태 조회 API shall 발급·전환 엔드포인트와 동일한 문서→워크스페이스 권한 게이트(member 이상, admin bypass)를 적용한다.

### Requirement 2: 로드 시 공유 상태 초기 판별 (프론트)
**Objective:** 공유 관리 권한자로서, 문서를 열 때 현재 공유 상태가 자동으로 반영되기를 원한다. 그래야 상태에 맞는 버튼을 즉시 볼 수 있다.

#### Acceptance Criteria
1. When 공유 관리 권한자가 공유 컨트롤이 노출되는 문서를 열면, the 문서 공유 컨트롤 shall 해당 문서의 현재 공유 상태를 조회해 반영한다.
2. While 공유 상태 조회가 진행 중이면, the 문서 공유 컨트롤 shall 조작 버튼을 확정된 공유/공유 해제 상태로 표기하지 않고 잠정(로딩·비활성) 상태로 표기한다.
3. If 공유 상태 조회가 실패하면, then the 문서 공유 컨트롤 shall 실패를 표면화하고 불확실한 상태를 공유 중으로 단정해 표기하지 않는다.
4. When 사용자가 선택 문서를 연속으로 전환하면, the 문서 공유 컨트롤 shall 가장 최근에 선택된 문서의 조회 결과만 반영한다(경합 방지).

### Requirement 3: 상태 기반 공유 컨트롤 노출 게이팅
**Objective:** 공유 관리 권한자로서, 적절한 권한·문맥에서만 공유 컨트롤이 보이기를 원한다. 그래야 권한 없는 사용자나 부적절한 문서에 공유 버튼이 노출되지 않는다.

#### Acceptance Criteria
1. While 현재 사용자가 owner 이거나 admin 이고, 현재 워크스페이스가 공유 가능(isShareable=true)하며, 대상 문서가 active 상태이면, the 문서 공유 컨트롤 shall 편집·삭제 컨트롤 옆에 공유 상태 기반 버튼을 노출한다.
2. If 현재 사용자가 owner 미만이고 admin도 아니면, then the 문서 공유 컨트롤 shall 공유 버튼을 노출하지 않는다.
3. If 현재 워크스페이스가 공유 불가(isShareable=false)이면, then the 문서 공유 컨트롤 shall 공유 버튼을 노출하지 않는다.
4. If 대상 문서가 active 상태가 아니면(예: 휴지통 문서), then the 문서 공유 컨트롤 shall 공유 버튼을 노출하지 않는다.
5. The 문서 공유 컨트롤 shall 프론트 노출 게이팅이 UX 편의일 뿐 서버측 권한 강제를 대체하지 않음을 전제로 동작한다.

### Requirement 4: 공유 / 공유 해제 동작 및 라벨 전환
**Objective:** 공유 관리 권한자로서, 하나의 버튼으로 공유를 켜고 끄고 싶다. 그래야 현재 상태를 간단히 전환할 수 있다.

#### Acceptance Criteria
1. While 대상 문서가 공유되지 않은 상태(공유 링크 없음 또는 링크 비활성)이면, the 문서 공유 컨트롤 shall 버튼 라벨을 "공유"로 표시한다.
2. While 대상 문서가 공유 중(공유 링크 활성)이면, the 문서 공유 컨트롤 shall 버튼 라벨을 "공유 해제"로 표시한다.
3. When 공유 관리 권한자가 "공유"를 실행하면, the 문서 공유 컨트롤 shall 대상 문서를 링크로 공개 접근 가능한 상태로 만들고 버튼을 "공유 해제"로 전환한다.
4. When 공유 관리 권한자가 "공유 해제"를 실행하면, the 문서 공유 컨트롤 shall 대상 문서의 공개 접근을 차단하고 버튼을 "공유"로 전환한다.
5. When 공유가 켜진 뒤 게스트가 해당 링크로 접근하면, the 공유 시스템 shall 기존 공개 렌더 경로로 문서를 읽기 전용으로 제공한다.
6. When 공유가 해제된 뒤 게스트가 이전 링크로 접근하면, the 공유 시스템 shall 접근을 404로 차단한다.

### Requirement 5: 공유 링크 복사
**Objective:** 공유 관리 권한자로서, 공유 중인 문서의 게스트용 링크를 한 번의 조작으로 복사하고 싶다. 그래야 링크를 게스트에게 전달할 수 있다.

#### Acceptance Criteria
1. While 대상 문서가 공유 중이면, the 문서 공유 컨트롤 shall "공유 해제" 버튼 옆에 "링크 복사" 버튼을 노출한다.
2. When 공유 관리 권한자가 "링크 복사"를 실행하면, the 문서 공유 컨트롤 shall 게스트용 공유 링크를 클립보드에 복사하고 복사 완료를 사용자에게 피드백한다.
3. If 클립보드 자동 복사가 실패하면(클립보드 API 부재·접근 거부), then the 문서 공유 컨트롤 shall 오류를 던지지 않고 사용자가 직접 선택·복사할 수 있는 링크 폴백을 노출한다.
4. The 문서 공유 컨트롤 shall 복사 대상으로 게스트용 프론트 링크를 사용하며 백엔드 공개 API 경로를 직접 사용하지 않는다.
5. While 대상 문서가 공유되지 않은 상태이면, the 문서 공유 컨트롤 shall "링크 복사" 버튼을 노출하지 않는다.

### Requirement 6: 오류 처리 및 상태 무결성
**Objective:** 공유 관리 권한자로서, 공유 조작 실패가 명확히 드러나고 화면 상태가 어긋나지 않기를 원한다. 그래야 실패를 인지하고 안전하게 재시도할 수 있다.

#### Acceptance Criteria
1. If 공유 또는 공유 해제 조작이 서버에서 실패하면(예: 공유 불가·비active로 인한 409), then the 문서 공유 컨트롤 shall 오류를 표면화하고 직전의 로컬 공유 상태를 훼손하지 않는다.
2. While 공유 또는 공유 해제 조작이 진행 중이면, the 문서 공유 컨트롤 shall 중복 실행을 막기 위해 조작 버튼을 비활성화한다.
3. The 문서 공유 컨트롤 shall 요청·인증·오류 정규화·전역 401 처리 등 교차 관심사를 공용 API 클라이언트에 위임하고 재구현하지 않는다.
